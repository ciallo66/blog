# MySql数据库

## 1.核心高频指令集
|操作目标|核心语法公式（直接套用）|2026 年全栈大白话翻译|
|:---|:---|:---|
|查库|SHOW DATABASES;|瞅瞅我这台电脑里一共有多少个数据库。|
|建库|CREATE DATABASE 库名;|新盖一个数据库房间。|
|进库|USE 库名;|切换到这个数据库，接下来全对它操作。|
|查表|SHOW TABLES;|看看当前数据库房间里，有哪些数据表。|
|查表结构|DESC 表名;|看看这张表里有哪些列（字段），长啥样。|
|增 (Create)|INSERT INTO 表名 (列1, 列2) VALUES (值1, 值2);|往表里塞进一条新数据。|
|查 (Read)|SELECT * FROM 表名 WHERE 条件;|把符合条件的数据全部捞出来看（*代表全选）|
|改 (Update)|UPDATE 表名 SET 列1 = 新值 WHERE 条件;| 必须带 WHERE，修改特定的一条数据。|
|删 (Delete)|DELETE FROM 表名 WHERE 条件;| 必须带 WHERE，删掉特定的一条数据。|


## 2.三种删除方式的区别
* **`DELETE FROM student WHERE id = '001';`**
  * **底层细节**：属于 DML 语句。逐行删除，**会写进 undo log 日志**。速度最慢，但支持撤回。如果不加 `WHERE`，会清空全表数据，只保留表头空壳。
* **`TRUNCATE TABLE student;`**
  * **底层细节**：属于 DDL 语句。直接把整张表连带索引全部删除，然后瞬间重建一张一模一样的空表。**不写日志**，速度极快，无法恢复！
* **`DROP TABLE student;` / `DROP DATABASE First;`**
  * **底层细节**：属于 DDL 语句。“删库跑路”操作，把数据、表头、索引、权限全部从硬盘上删除！

## 3.数据类型的操作

###  1. 文本/字符串类型表
| 数据类型 | 加括号 `(N)` | 功能场景 | 核心细节 |
| :--- | :---: | :--- | :--- |
| **`VARCHAR(N)`** | **是**  | 用户名、手机号、密码 | **变长文本**。`N`为最大字符数（上限65535）。按实际长度+长度头存，最省硬盘。 |
| **`CHAR(N)`** | 否(默1) | 身份证号、性别、固定编码 | **定长文本**。死死占用 `N` 字符空间，不满补空格。读写速度略快。 |
| **`TEXT`** | **否**  | 博客文章、商品详情描述 | **超长文本**。无需指定长度。专吞海量文字，不占行内最大生命值。 |

###  2. 数字类型表
| 数据类型 | 加括号 `(N)` | 功能场景 | 核心细节 |
| :--- | :---: | :--- | :--- |
| **`INT`** | **否**  | 年龄、商品库存、点赞数 | **标准整数**。固定4字节，上限正负21亿。括号数字已作废，别写。 |
| **`BIGINT`** | **否**  | 雪花ID、电商订单号 | **大整数**。固定8字节，上限 $9\times10^{18}$。长ID和订单号防爆仓专用。 |
| **`TINYINT`** | **否**  | 状态值(0/1)、布尔映射 | **微整数**。固定1字节（-128~127）。大厂最爱用作状态开关，极省空间。 |
| **`DECIMAL(M,D)`**| **是**  | **商品价格、银行余额(钱)** | **高精小数**。底层用字符串模拟，**绝无二进制精度丢失**。M总位，D小数位。 |

###  3. 时间/日期类型表
| 数据类型 | 加括号 `(N)` | 功能场景 | 核心细节 |
| :--- | :---: | :--- | :--- |
| **`DATETIME`** | **否**  | 订单创建时间、注册时间 | **年月日 时分秒**。格式为 `2026-05-19 17:00:00`，范围1000-9999年。 |
| **`DATE`** | **否**  | 员工入职日期、生日 | **纯年月日**。砍掉时分秒，格式为 `2026-05-19`。不需精细到秒的业务用。 |


> 当未来使用 Node.js 连接 MySql 插入 DATETIME 时，Node 驱动默认会转换成 UTC 时区，导致存入的时间永远比北京时间少 8 个小时！
> **解法**：在配置里显式加上 `timezone: '+08:00'` 锁死东八区

## 4.约束
|约束类型|是否允许为空|	是否允许重复	|表中数量	|核心用途|
|-|-|-|-|-|
|PRIMARY |KEY	❌ 不允许	|❌ 不允许|	1 个	|唯一标识行|
|UNIQUE |KEY	✅ 允许（多个 NULL 通常视为不同）|	❌ 不允许|	多个	|保证列值唯一|
|CHECK	|看字段定义	|看条件	|多个	|限制数据范围 / 逻辑|

## 
| 概念 | 核心作用 |
|------|----------|
| **列** | 数据的基本单元，定义字段类型、默认值及是否可空 |
| **约束** | 保证数据的合法性和完整性（NOT NULL、UNIQUE、CHECK等） |
| **外键** | 维护跨表引用完整性，支持级联操作 |
| **引用** | 表间关系，连接父表与子表，形成实体依赖 |
| **触发器** | 在数据变更时自动执行逻辑（审计、校验、级联更新） |
| **索引** | 加速查询、排序和分组，实现唯一性约束 |
| **分区** | 水平拆分大表，优化管理、查询和维护效率 |
| **Statistics** | 查询优化器依此选择最佳执行计划 |
| **DDL** | 定义和修改数据库结构的语言（CREATE/ALTER/DROP） |
| **虚拟** | 基于表达式或查询的逻辑结构（虚拟列、视图），减少冗余 |

```sql
SELECT tags.id, COUNT(question_tags.question_id)
FROM tags
LEFT JOIN question_tags ON tags.id = question_tags.tag_id

WHERE tags.status = 'active'             -- 1. [WHERE] 先把封禁的标签踢掉
GROUP BY tags.id                         -- 2. [GROUP BY] 把剩下的按标签ID揉成堆
HAVING COUNT(question_tags.question_id) > 0 -- 3. [HAVING] 只要有帖子的堆，0个帖子的不要
ORDER BY tags.name                       -- 4. [ORDER BY] 剩下来的堆按字母排好队
LIMIT 10 OFFSET 0;                       -- 5. [LIMIT/OFFSET] 最后切出前10条吐给前端
```

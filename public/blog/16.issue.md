# 前后端连接所遇到的问题

## 1.后端cors处理跨域问题
### 初始化cors
```js
//1.安装cors
npm i cors

//2.导入cors
const cors = require('cors')

//3.运行cors
app.use(cors());
```
### 传参数
>常见参数

|参数	      |作用                      |	示例            | 
|------------ | ----------------------- | -------------------------------|
|origin       | 允许哪个前端域名访问你的后端(可传数组函数)|{ origin: 'http://localhost:8080' }|
|methods      |	允许的 HTTP 方法        |	['GET','POST','PUT','DELETE']   |
|allowedHeaders|	允许的自定义请求头   |	['Content-Type','Authorization']  |
|credentials  |	是否允许携带 Cookie    |	true（需配合前端 withCredentials）  |
|maxAge       |预检请求的有效期（秒）  |	86400（24小时）                      |


**注意事项**:
- cors()不传参数时默认允许所有来源，即origin:'*'


## 2. mysql2里sql.query()
### (1)**query完整格式**:
**老方法**:sql.query(sql, values, callback)  (老方法)  
**新方法**:const [results,fields] = await sql.query(sql ,valuse)

| 参数位置 | 名称 | 类型 | 必填 | 说明 |
|----------|------|------|------|------|
| 第一个 | sql | string | 是 | 要执行的 SQL 语句，使用 `?` 作为参数占位符 |
| 第二个 | values | array / 单个值 | 否 | 替换占位符的实际值，数组（多个参数）或单个值（一个参数） |
| 第三个 | callback | function(error, results, fields) | 是 | 回调函数，执行完成后的处理 |


### (2)**callback完整格式格式**:
**老方法**:function(error, results, fields)
**新方法**:无回调函数,err放在了catch(err),执行完成返回一个包含results和fields的数组
>注意:因为是数组不能用{}解构赋值,常用const [results] = await sql.query(sql ,valuse)

| 参数 | 说明 |
|------|------|
| error | 如果 SQL 执行出错，这里是一个 Error 对象；成功则为 null |
| results | 查询结果。SELECT 返回数组，写操作返回 OkPacket 对象 |
| fields | 列信息数组（很少用到） |

### (3)results
**注意**:当使用SELECT查询的时候是没有以下操作的，results就是获得到的数据，直接输出
#### 读操作：
当执行 SELECT 时，results 是一个数组（即使没有数据也是空数组 []），每个元素是一个对象，代表一行。  
>results.length → 判断有没有数据  
>results[0] → 拿到第一个对象（查单个用户时）  

#### 写操作：
当执行 INSERT、UPDATE、DELETE 时，results 是一个对象（通常称为 OkPacket 或 ResultSetHeader）  
>result.affectedRows → 判断操作是否成功（如果为 0 说明没改到任何行）  
>result.insertId → 获取新插入记录的 ID（INSERT 时）  





## 3. 网线两端的 JSON 秘密
* **前后端网线隔离**：前端用 Axios 还是 Fetch，和后端用不用 `async/await` 查数据库 **100% 独立隔离**，都为了方便
* **后端发送**：后端无法把 Promise 对象扔进网线（网线只跑纯文本字符串）。后端必须在本地用 `await` 解构赋值等把数据捞成真实数组，再通过 `res.send()` 变文本发走。
* **前端转 JSON 的规矩**：
  * 后端执行 `res.send()` 或 `res.json()` 时，底层已**自动**把对象转成 JSON 字符串发出了。
  * 前端收到后：用 **Axios** 会自动变回对象直接用；用原生 **Fetch** 必须手动写一句 `await response.json()` 进行手动解码。

# 📂 前端发送数据的三大核心报文格式与后端解析对比表

当前端向后端发送数据（主要在 POST / PUT / PATCH 请求的 Body 中）时，网线里流动的全是纯文本字符串。由于数据的组织形式不同，前端需要通过请求头（Content-Type）为报文贴上格式标签，后端 Express 则通过对应的中间件或插件将其“复活”为 JavaScript 对象。

| 数据格式名称 | 前端报文标签 <br>`(Content-Type)` | 网线中流动的纯文本长相（示例） | 后端 Express 复活机制 <br>*(解析到 `req.body`)* | 核心适用场景 |
| :--- | :--- | :--- | :--- | :--- |
| **JSON 格式** <br>*(现代化主流)* | `application/json` | `'{"name":"张三","age":18}'` | **全局中间件**：<br>`app.use(express.json());`<br>*(底层自动执行 `JSON.parse`)* | 目前 90% 以上的前后端分离项目、Vue/React 与 Node.js 的标准数据交互。 |
| **传统表单格式** <br>*(URL Encoded)* | `application/x-www-form-urlencoded` | `'name=%E5%BC%A0%E4%B8%89&age=18'`<br>*(注：汉字会被浏览器自动进行 URL 百分号编码)* | **全局中间件**：<br>`app.use(express.urlencoded({ extended: false }));`<br>*(底层自动解码并拆分键值对)* | 传统的 HTML `<form>` 表单提交、部分老旧系统的接口对接。 |
| **多部分/媒体文件格式** <br>*(Multipart 表单)* | `multipart/form-data; boundary=----WebKitFormBoundary...`<br>*(注：`boundary` 是浏览器生成的货物隔离板标签)* | 一串极其复杂的混合文本流。<br>通过隔离板字符串，把**普通文本**和**被编码成文本形式的二进制文件（如图片）**一段段隔开。 | **Express 自带中间件无法解析**。<br>必须引入第三方解析插件（如 **`multer` 包**）。<br>解析后，文本归 `req.body`，文件归 **`req.file`**。 | 用户头像上传、Excel/PDF 附件上传、音视频资源上行。 |

>  **全栈核心认知**：
> * 前端传数据完全由你使用的前端工具决定（**Axios** 传对象会自动变 JSON 文本并贴标签；原生 **Fetch** 必须手动写 `JSON.stringify` 变成字符串并手动配 Headers）。
> * 后端哪怕路由再多，只要在顶部配置好对应的 `app.use()` 中间件，海关就会根据前端请求头里的 `Content-Type` 标签“对号入座”自动解析，转变成一个可使用的js对象

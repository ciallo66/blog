## 二.JWT（JSON Web Token）
### (1)介绍
**JWT是一个概念**:用三段式字符串、不查数据库、纯靠密钥验真的通行证安全机制
**作用**:常用于验证用户的信息，比如用户去发表评论，去验证token,对应的账户的数据库就有这条评论  
#### token由三部分组成：A.B.C（即 Header.Payload.Signature）。
- 第一段（Header 头部）：写着基本信息，比如：“我是 JWT，我底层用的加密算法是 HS256”。

- 第二段（Payload 载荷 —— 最重要）：这里面装的是用户的非敏感数值。比如用户的 userId、username、还有这张票什么时候过期（exp）。

- 第三段（Signature 签名 —— 安全核心）：这是后端用自己的“绝密密钥（Secret Key）”，把前两段拼在一起算出来的一串哈希值。它是用来防伪的,别人想要修改没有对应上密钥就无法修改

### (2)写法
#### 1.初始化
```js
//安装当前最好用的jsonwebtoken库
npm install jsonwebtoken
//导入
const jwt = require('jsonwebtoken');
//后端里的绝密秘钥，用于防伪，一般写在.env配置环境里
const MY_SECRET_KEY = 'gemen_ni_tai_niu_bi_le_123456';
//.env格式
MY_SECRET_KEY = gemen_ni_tai_niu_bi_le_123456
```
#### 2.发证(jwt.sign())
```js
//传参顺序必须:载荷 -> 密钥 -> 头部  ,因为必填参数必须放在前面，头部是选填的
function 创建通行证(userId, username) {
  const token = jwt.sign(
    { id: userId, name: username }, 
    MY_SECRET_KEY, 
    { expiresIn: '2h' } 
  );
  return token; // 返回那串 aaaaa.bbbbb.ccccc 的三段式字符串
}
```
#### 3.验证(jwt.verify())
```js
function 检查通行证(前端传来的Token) {
  try {
    const 解密后的数值 = jwt.verify(前端传来的Token, MY_SECRET_KEY);
    
    // 如果是真的，返回数据：{ id: 24, name: '大锤', iat: xxx, exp: xxx }
    return { isValid: true, payload: decoded };
  } catch (err) {
    //  如果被篡改了，或者时间过期了，verify 直接进 catch
    return { isValid: false, reason: 'Invalid or expired token' };
  }
}
```

### (3)注意事项
1. 载荷（Payload）里不能放敏感信息（比如密码），它没有任何加密
2. 密钥（Secret）强度一定要够，太简单容易泄露
3. 过期时间（expiresIn）要合理(2h 7d)

## 三.数据库的预处理(在后端叫参数化查询，防SQL注入)
** 作用** :在利用 `mysql2` 连接池操作数据库时，为了彻底杜绝 **SQL 注入（SQL Injection）** 漏洞，必须严格遵循**预处理（Prepared Statements）**的格式规范。

**原理**:
1. 机制：预处理的核心优势在于“代码与数据分离”。送入 ? 的数据，在 MySQL 内部绝不会被当成 SQL 命令去执行。
2. 效果：即使黑客在前端恶意输入 ' OR '1'='1，MySQL 也只会把它当成一个普通的“纯文本字符串”去匹配，从而保证数据库安全。

### (1)标准预处理语法格式

标准的预处理执行通过 `pool.execute()` 或 `db.query()` 传入**两个核心参数**：

```javascript
const [rows] = await pool.execute(
  'SELECT * FROM users WHERE username = ? AND status = ?', // 参数 1：带占位符的 SQL 模板
  [dbSearchKey, userStatus]                                // 参数 2：按顺序排列的变量数组
);
```
>参数 1（纯净模板）：SQL 语句中所有动态变化的变量，一律用 ?（问号占位符） 代替。此时语句仅作为“命令结构”发送给数据库,不直接执行  
>参数 2（变量数组）：必须用 []（方括号） 将变量包裹成数组。数组内的变量会严格按照从左到右的顺序，一一对应替换模板中的 ?

### (2)注意事项
1. 最好不要使用 username = '${dbSearchKey}'` ,有SQL风险
2. `[]`必不可少，必须用数组格式
3. 有几个 ？ ，数组里面就要有几个数据，顺序也要严格对照

## 五.bcrypt(把密码加密如数据库)
### (1)介绍
#### bcrypt 是一个专门为密码量身定制的单向不可逆哈希算法。
#### 特点：
1. 单向不可逆：可以把“明文密码”丢进去绞成“一堆乱码”，没有任何人和算法还原出 明文密码
2. 自带盐（Salt）机制：即使两个用户的密码一模一样，bcrypt 加密出来的两串乱码也完全不同

### (2)使用
#### 1.初始化
```js
//安装包
npm install bcrypt
//导入包
const bcrypt = require('bcrypt');
```

#### 2.注册时  bcrypt.hash(明文, 轮数)
```js
const 乱码密码 = await bcrypt.hash('123', 10); 
console.log(乱码密码); // 打印出来长这样：$2b$10$EixZZ... (这就存进数据库了)
```

#### 3.登录时  bcrypt.compare(明文, 密文)
```js
const 对上没有 = await bcrypt.compare('123', 乱码密码);
console.log(对上没有); // 如果输入正确，当场打印：true！放行！
```

### (3)注意事项
1. 绝对不要用同步写法，非常吃 CPU 资源，需要使用异步
2. 数据库密码字段的长度至少 60 位以上，因为bcrypt 加密出来的密文哈希值永远是固定的 60 位字符
3. 盐的轮数（Salt Rounds）不要盲目追求过高，推荐SALT_ROUNDS = 10 代表加密的复杂度是 2 的 10 次方
>Salt（盐）： 往密码里加一串随机数，让相同的密码变成不同的哈希值，破掉黑客的对照表。
>Salt Rounds（盐轮数）： 决定把这个加密过程循环嵌套算多少次，让黑客难易解析。

## 七.正则表达式（简称 Regex 或 RE）
### (1)概念
它就是一个超强版、带模糊匹配的“字符串安检机”和“文本替换大杀器”
**作用**：
1. 数据校验（校验电话号码是不是11位或开头是不是1）
2. 文本提取（捞所有的jpg文件 http://...jpg）
3. 文本魔改替换（# 标题 换成 <h2>标题</h2>）

### (2)写法(常用符号)
| 分类 | 符号 | 符号名称 | 大白话释义 | 经典实战举例 |
| :--- | :---: | :--- | :--- | :--- |
| **字符指挥官**<br>*(找什么类型的字)* | **`\d`** | 数字字符 | 匹配 0 到 9 的**任意单个数字** (Digital) | `/^\d{6}$/` <br>*(校验 6 位纯数字支付密码)* |
| | **`\D`** | 非数字字符 | 匹配任意一个**不是数字**的字符 (大写代表反义) | 过滤掉文本里所有的数字 |
| | **`\w`** | 单词字符 | 匹配**字母、数字或下划线** (Word) | `/\w+/` <br>*(匹配用户名或代码语言名)* |
| | **`\s`** | 空白字符 | 匹配**空格、制表符(Tab)或换行符** (Space) | `/\s+/` <br>*(去除文本里多余的空格)* |
| | **`.`** | 万能匹配点 | 匹配除换行符之外的**任意单个字符** | `/a.c/` <br>*(能匹配 abc、axc、a8c 等)* |
| **数量工头**<br>*(要找多少个)* | **`*`** | 星号（任意个） | 代表前面的字符有或者没有，或者有很多个 **(0次到无数次)** | `/a*/` <br>*(匹配空字符串，或者 a、aa、aaa...)* |
| | **`+`** | 加号（至少一个）| 代表前面的字符**至少得有一个**，上不封顶 **(1次到无数次)** | `/\d+/` <br>*(连续的一串数字，如版本号 `123`)* |
| | **`?`** | 问号（可有可无）| 代表前面的字符**最多只能出现一次** **(0次或1次)** | `/https?/` <br>*(兼容匹配 `http` 和 `https`)* |
| | **`{n}`** | 精确数量限定 | 代表前面的字符**必须连续出现刚好 n 次** | `/\d{11}/` <br>*(精确匹配 11 位国内手机号)* |
| **边界警卫**<br>*(在哪里找才算数)*| **`^`** | 脱字符（行首） | 表示必须在**一行的开头**看到后面的字符 | `/^# /` <br>*(必须以 `#` 开头，常用于匹配 MD 标题)* |
| | **`$`** | 美元符（行尾） | 表示必须在**一行的末尾**看到前面的字符 | `/\.js$/` <br>*(判断文件名是不是以 `.js` 结尾)* |
| **高级特工**<br>*(分组与魔改)* | **`()`** | 小括号（分组捕捉）| 把一部分规则**打包成一个整体**，并把里面捞到的“核心肉”抓出来赋予代号 `$1`, `$2` | `/^## (.+)$/` <br>*(把 `## ` 后面的标题文字抓出来变成 `$1`)* |
| | **`\|`** | 竖线（或条件） | 代表**或者**，左右两边的规则只要满足一个就行 | `/zh-CN\|en-US/` <br>*(匹配中文或英文语系)* |

### (3)注意事项
1. 难理解，写完要加上注释
2. 不要嵌套使用，ReDoS容易卡死界面
3. 复杂逻辑少自己写，容易漏东西

## 八.Zod库(用于检验数据，Zod适用于js,ts)
### (1)作用
1. **当安检门**：在**代码运行的时候**，强行把所有类型不对、格式错乱的脏数据（前端表单、接口请求）**拦在系统外面**，防止程序崩溃。
2. **过滤垃圾**：自动把数据里**多余的、未定义的恶意字段直接擦除剔除**（`strip` 功能），只留下你需要的数据，防数据库污染。
3. **白嫖类型**：你写一份 Zod 校验规则，就能**一键自动生成对应的 TypeScript 类型**（`z.infer`），不用再手写一遍 `interface`。
4. **自动报数**：校验失败时，它会**自动吐出精准的错误位置和原因**（哪个字段错、为什么错），彻底省掉你手写 `if-else` 拼错误提示的垃圾时间。

### (2)写法
#### 1. 核心安检与类型功能
| 功能方法 | 核心作用 | 实际表现 |
| :--- | :--- | :--- |
| `.strip()` | 自动擦除未定义的脏字段 | 默认规则。只要多传了 Schema 里没写的字段，直接抠掉抹去，不影响后续代码。 |
| `.passthrough()` | 允许并原样保留未定义字段 | 允许前端多传字段，并且多传的字段不会被删掉，会原样透传过去。 |
| `.strict()` | 发现多余字段直接报错拦截 | 严厉模式。只要前端多传了任何一个没定义的字段，直接当场砸饭碗报错。 |
| `z.infer<typeof Schema>` | 一键逆向推导 TS 类型 | 核心功能。通过这行代码直接让 TS 自动生成对应的 type 提示，免手写 interface。 |
| `.safeParse(data)` | 温和校验数据 | 常用运行校验。失败时只返回错误原因，绝对不会让程序崩溃终止。 |

#### 2. 数据类型语法模板

| 数据类型 | 基础模板 | 可选连缀方法（按需自由组合） | 核心限制与坑点备注 |
| :--- | :--- | :--- | :--- |
| **数字 (Number)** | `z.number()` | `.int()`【限制整数】<br>`.gt(大于)`<br>`.gte(大于等于)`<br>`.lt(小于)`<br>`.lte(小于等于)` | ⚠️ **Zod 官方无内置的小数限制方法**！<br>若要强行卡死最多两位小数，必须挂载自定义正则：<br>`.refine(val => /^\d+(\.\d{1,2})?$/.test(val.toString()), { message: "最多两位小数" })` |
| **对象 (Object)** | `z.object({ 键: 类型 })` | `.strip()`【可选】<br>`.passthrough()`【可选】<br>`.strict()`【可选】 | 最常用的接口/表单容器。基础模板定义好后，必须在末尾按需连缀上述三个方法之一来决定如何处理多余字段。 |
| **字符串 (String)** | `z.string()` | `.min(最小长度)`【可选】<br>`.max(最大长度)`【可选】<br>`.email()`【限制邮箱】<br>`.url()`【限制网址】 | 默认拒绝 `null` 和 `undefined`。内置了常用的邮箱和网址正则格式限制，直接点出来用。 |
| **数组 (Array)** | `z.array(Zod类型)` | `.min(最小长度)`【可选】<br>`.max(最大长度)`【可选】 | 括号里必须传入具体子项的 Zod 类型（例如限制字符串数组：`z.array(z.string())`）。 |


#### 3. 修饰符模板（全局通用后缀）

| 修饰方法 | 语法模板 | 允许的值 | 实际作用 |
| :--- | :--- | :--- | :--- |
| **可选** | `Zod类型.optional()` | 对应类型 或 `undefined` | 允许前端不传这个字段，如果不传，代码里拿到的就是 `undefined`。 |
| **可空** | `Zod类型.nullable()` | 对应类型 或 `null` | 前端必须传这个字段，但是字段的值允许传 `null`。 |

### (3)注意事项
1. 接口防崩
业务代码里一律用 **`.safeParse(data)`**。用 `.parse()` 一旦报错会直接导致进程或页面崩溃。

2. 脏数据拦截
定义对象结构时，末尾必须连缀 **`.strip()`**。不加的话，前端多传的未知字段会被默认放行。

3. 金额/小数校验
Zod 官方没有内置限制小数的方法。要卡死两位小数（如金额），必须手写 **`.refine()`** 自定义正则：
  `.refine(val => /^\d+(\.\d{1,2})?$/.test(val.toString()))`

## 十.数据库联表查询
### (1)介绍
把两张表有关系的行，用WHERE条件连接起来，变成一张大表。
### (2)写法
**格式**:
```sql
SELECT questions.id, questions.title, users.username 
FROM questions
INNER JOIN users ON questions.user_id = users.id;
```

1. INNER JOIN（内连接，最常用 🌟）：

大白话：两边必须都有才能拼。如果一个问题没有对应的作者，或者一个用户没提过问，两边对不上，那这行数据直接扔掉，不打印。

2. LEFT JOIN（左外连接 🌟）：

大白话：以左边的表（FROM 后面的表）为主！ 就算左边的问题在右边找不到作者，问题也必须打印出来，作者名字那一栏直接显示 NULL（空）。在社区项目里，为了防止文章因为作者账号注销而直接消失，普遍用 LEFT JOIN。

3. RIGHT JOIN（右外连接）：

大白话：和左连接反过来，以右边的表为主。因为调换一下表的顺序就能用 LEFT JOIN 代替，所以实际开发里很少有人用它。


### (3)注意事项
1. 别忘了写 ON 后面的条件，不写卡死
2. 字段名冲突时要加“表名作前缀”(如questions.id和users.id)
3. 不要疯狂连太多张表,最多3,4张，太多容易卡顿

## 十一.数据库多对多
### 1. 创建一个中间表
* **场景**：一个【提问】可贴多个【标签】，一个【标签】可包含多个【提问】。
* **解决**：不能改主表，必须建第三张表——**中间表（如 `question_tags`）**。
* **本质**：不存文案，只存两个主表的 `ID` 对。用中间表的外键链接主表的主键ID。

### 2. 核心关系：发起者与接收者
> **黄金总结**：“外键是发起者（小弟），引用是接收者（老大）。A连B，A有外键，B有引用。”

| 角色 | 对应表 | 核心约束 | 职责大白话 |
| :--- | :--- | :--- | :--- |
| **接收者** (老大) | `questions` / `tags` (主表) | **主键约束** (`PRIMARY KEY`) | 保证自己ID独一无二、不是黑户，给外键提供精准引用的目标。 |
| **发起者** (小弟) | `question_tags` (中间表) | **外键约束** (`FOREIGN KEY`) | 主动牵出钢丝线，死死绑定并**引用**主表的主键 ID。 |

### 3. 联动规则（删除与更新）
* **删除时：级联（CASCADE）** *(推荐)*：删掉某个问题，中间表对应的连线行**全自动跟着删掉**，不留垃圾数据。
* **更新时：限制（RESTRICT）**：小弟牵着线，老大别想跑。中间表还连着某个标签时，**拒绝**任何人修改主表该标签的 ID。

### 4. 数据如何流动？
* **增（多加标签）**：在中间表**多生出几行**。先插问题表拿到新 `question_id`（如 3），再循环标签 ID 数组（如 `[1,2,3]`），往中间表批量插 3 行：`(3,1)`, `(3,2)`, `(3,3)`。
* **查（连表查询）**：顺着中间表搭的桥，连续两次 `JOIN` 摸藤摘瓜：
```sql
SELECT questions.title, tags.name
FROM questions
INNER JOIN question_tags ON questions.id = question_tags.question_id -- 第一步：连上红娘
INNER JOIN tags ON question_tags.tag_id = tags.id;                  -- 第二步：连上标签
```

## 十三.compression压缩(性能优化)
在前端工程化部署中，对打包后的静态资源（HTML、JS、CSS）进行 **Gzip** 或 **Brotli** 压缩，是提升页面首屏加载速度（FCP/LCP）最丝滑、见效最快的手法。

### 1. 为什么必须做传输压缩？

1. **体积降维打击**：文本文件（如 JS、CSS）的压缩率通常可达 **70% ~ 80%**。原本 1MB 的大 JS 文件，压缩后可能只有 200KB 左右。
2. **省带宽与网络开销**：传输体积暴降，意味着网络延迟（TTFB）大幅缩短，同时能为服务器节省海量带宽流量。
3. **Brotli > Gzip**：Brotli 是 Google 推出的新一代压缩算法，在相同文件大小下，压缩率比 Gzip 还要高出 **15% ~ 25%**，目前主流浏览器均已完美支持。


### 2. 核心原理：动态压缩 vs 静态预压缩

在实际部署中，通常有两种实现方案。为了追求极致性能，**强烈推荐使用“静态预压缩”**。

| 方案 | 原理 | 优点 | 缺点 |
| :--- | :--- | :--- | :--- |
| **动态压缩**<br>(On-the-fly) | 浏览器请求资源时，Nginx **临时**在内存中对文件进行压缩，然后返回。 | 无需前端配置，服务端开箱即用。 | **极度消耗服务器 CPU**！高并发时，服务器容易卡死在压缩性能上。 |
| **静态预压缩**<br>(Pre-compressed) | 前端打包时（Vite/Webpack）直接生成 `.gz` 或 `.br` 文件；Nginx 收到请求后，**直接读取**现成文件返回。 | **服务器 CPU 零消耗**，响应速度达到极致。 | 需要前端配置插件，且打包后的产物占用稍多一点磁盘空间。 |



### 3.前端配置：Vite 自动化打包生成压缩包

在 Vue3/Vite 项目中，使用 `vite-plugin-compression` 插件，可以在执行 `npm run build` 时自动榨干文件体积。

#### 1. 安装插件
```bash
npm i vite-plugin-compression -D
```
